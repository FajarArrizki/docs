The model layer is still important, but it is no longer the whole billing story.

Rabit uses the OpenRouter model manager in the codebase:

- `agents/openrouter/models.py`
- `agents/openrouter/database.py`
- `data/openrouter_models.json`

## What the model layer is for

| Concern | Why it matters |
| --- | --- |
| model catalog | the backend needs to know which models are available right now |
| pricing metadata | model pricing drives `model_cost_usd` estimation |
| capability metadata | tools, reasoning, and context length matter for product behavior |
| local cache | the product should not depend on a fresh upstream call every time |

## How model cost works

```mermaid
flowchart LR
    A[Intent router call] --> D[OpenRouter session cost ledger]
    B[Main response call] --> D
    C[Tool follow-up or compression call] --> D
    D --> E[model_cost_usd]
    D --> F[/api/openrouter/session-costs/{scope_id}]
```

## Metadata the backend tracks

| Field | Why it matters |
| --- | --- |
| `id` and `name` | stable model identity for backend and client surfaces |
| `provider` | vendor grouping |
| `context_length` | prompt-window limits |
| `input_price` and `output_price` | cost estimation |
| `supports_tools` | tool-calling compatibility |
| `supports_reasoning` | reasoning-capable filtering |
| `enabled` | backend-side model availability control |

## Why this page matters

The model catalog explains `which models exist` and `how model pricing is estimated`.

It does not explain monitoring cost.

That is why Rabit now treats model cost as one part of a broader service-cost story.
