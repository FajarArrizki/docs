## F. Treasury / Fee Recipient Rotation

**Current**: Fee vault is PDA (cannot be rotated)

**Rationale**:
- PDA ensures security (no private key)
- Admin claims fees via `claim_fees`
- Admin can transfer authority to rotate recipient

**Alternative** (future enhancement):
- Add `fee_recipient` field to config
- Transfer fees to recipient instead of PDA
- Allows rotation without authority transfer

## G. Delegation Expiry

**Current Implementation**:
- User sets `expiry_duration` on creation
- Checked on every usage: `!delegated_signer.is_expired(current_time)`
- Instant rejection if expired

**Monitoring**:
- Backend should check expiry before submitting
- Users should monitor expiry and renew if needed

**Future Enhancement**:
- Add `extend_delegation` instruction
- Allow extending expiry without recreating

## H. Replay / Duplicate Protection

### H.1 Usage Record Deduplication

**Current Protection**:
- PDA seeds include timestamp: `[AI_USAGE_SEED, vault.key(), timestamp.to_le_bytes()]`
- Prevents duplicates within same second
- Init constraint prevents duplicate PDA

**Limitation**:
- Backend can submit multiple records in different seconds

**Mitigation**:
- Backend must implement off-chain deduplication
- Usage records are immutable (audit trail)

**Future Enhancement**:
- Add sequence number to PDA seed
- Add nonce field to usage record

---

### H.2 Transaction Replay Protection

**Protection**: Solana's recent blockhash mechanism
- Transactions expire after ~2 minutes
- Cannot replay old transactions

## K. Migration / Upgrade Considerations

### K.1 Program Upgrade Strategy

**Solana Program Upgrades**:
- Program is upgradeable (default Anchor behavior)
- Upgrade authority = deployer wallet
- Can deploy new program version without changing program ID

**Account Compatibility**:
- Existing accounts remain valid after upgrade
- New fields can be added (requires migration)
- Existing fields cannot be removed (breaking change)

### K.2 State Migration

**Adding New Fields**:
1. Add field to struct with default value
2. Deploy new program version
3. Existing accounts work (field = default)
4. New accounts include new field

**Removing Fields** (Breaking Change):
1. Create new account type with different discriminator
2. Create migration instruction
3. Users migrate old accounts to new accounts
4. Deprecate old account type

**Changing Field Types** (Breaking Change):
1. Same as removing fields
2. Requires full migration

### K.3 Upgrade Checklist

**Pre-Upgrade**:
- [ ] Test new program on devnet
- [ ] Verify account compatibility
- [ ] Prepare migration instructions (if needed)
- [ ] Notify users of upgrade
- [ ] Backup program state

**Upgrade**:
- [ ] Deploy new program version
- [ ] Verify deployment successful
- [ ] Test all instructions
- [ ] Monitor for errors

**Post-Upgrade**:
- [ ] Verify all accounts work
- [ ] Run migration (if needed)
- [ ] Monitor usage for issues
- [ ] Document changes

### K.4 Backward Compatibility

**Maintaining Compatibility**:
- Add new instructions (safe)
- Add optional parameters (safe)
- Add new fields with defaults (safe)
- Emit new events (safe)

**Breaking Changes**:
- Remove instructions (breaks clients)
- Remove parameters (breaks clients)
- Remove fields (breaks accounts)
- Change PDA seeds (breaks accounts)
