## 1. Admin Abuse

### Threat 1.1: Fee Manipulation
**Attack**: Admin increases platform_fee_bps to 100% (10000 bps) to drain user funds

**Current Protection**:
-  `MAX_PLATFORM_FEE_BPS = 2000` (20% max)
-  `require!(new_platform_fee_bps <= MAX_PLATFORM_FEE_BPS)`

**Residual Risk**: Admin can still set 20% fee (high but not catastrophic)

**Mitigation**:
- Document fee limits in user agreement
- Implement timelock for fee changes (future enhancement)
- Multi-sig for admin authority (deployment best practice)

**Severity**: Medium (limited by MAX_FEE)

---

### Threat 1.2: Unauthorized Fee Claiming
**Attack**: Non-admin tries to claim fees from fee_vault

**Current Protection**:
-  `has_one = authority` constraint on ClaimFees context
-  Only config.authority can sign

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 1.3: Authority Takeover
**Attack**: Admin transfers authority to attacker-controlled wallet

**Current Protection**:
-  Requires current authority signature
-  No way to bypass authority check

**Residual Risk**: Admin key compromise = full platform compromise

**Mitigation**:
- Use hardware wallet for admin key
- Implement multi-sig (e.g., Squads)
- Implement timelock for authority transfer (future)
- Monitor authority changes via events

**Severity**: Critical (requires admin key compromise)

---

### Threat 1.4: Malicious Pause
**Attack**: Admin pauses platform indefinitely to DoS users

**Current Protection**:
-  Pause only affects AI usage recording
-  Users can still deposit/withdraw from vaults
-  Users can still revoke delegations

**Residual Risk**: Users cannot use AI services while paused

**Mitigation**:
- Document pause policy
- Implement automatic unpause after timeout (future)
- Multi-sig for pause action

**Severity**: Medium (limited impact, users retain fund access)

---

### Threat 1.5: Backend Authority Abuse
**Attack**: Admin sets malicious backend_authority that submits fake usage

**Current Protection**:
- Warning:  backend_authority is informational only (not enforced)
-  Delegations are per-user (user chooses who to delegate to)

**Residual Risk**: If user delegates to malicious backend, backend can drain vault up to spending limit

**Mitigation**:
- Users should only delegate to trusted backends
- Spending limits cap maximum damage
- Short expiry reduces exposure window
- Enforce backend_authority in delegation usage (future enhancement)

**Severity**: Medium (requires user to delegate to malicious backend)

---

## 2. Unauthorized Delegation Use

### Threat 2.1: Delegation Hijacking
**Attack**: Attacker tries to use someone else's delegation

**Current Protection**:
-  PDA seeds include both owner and delegate pubkey
-  `constraint = delegated_signer.delegated_pubkey == delegate.key()`
-  Delegate must sign transaction

**Residual Risk**: None (cryptographically impossible)

**Severity**: None

---

### Threat 2.2: Expired Delegation Use
**Attack**: Backend continues using delegation after expiry

**Current Protection**:
-  `require!(!delegated_signer.is_expired(clock.unix_timestamp))`
-  Checked on every usage

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 2.3: Revoked Delegation Use
**Attack**: Backend uses delegation after user revokes it

**Current Protection**:
-  `require!(delegated_signer.is_active)`
-  Checked on every usage
-  Revocation is instant

**Residual Risk**: Race condition - in-flight transaction may land after revocation

**Mitigation**:
- Backend should check delegation status before submitting
- Users should monitor for unauthorized usage after revocation
- Spending limits cap maximum damage

**Severity**: Low (race condition window is small, limited by spending limit)

---

### Threat 2.4: Spending Limit Bypass
**Attack**: Backend tries to exceed spending limit

**Current Protection**:
-  `require!(delegated_signer.can_spend(total_charged))`
-  `delegated_signer.spent_amount += total_charged` after each usage
-  Checked before charge

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 2.5: Delegation Frontrunning
**Attack**: Attacker observes user creating delegation, frontruns to create delegation with same seeds

**Current Protection**:
-  PDA seeds include owner pubkey (attacker cannot use victim's pubkey)
-  Init constraint prevents duplicate PDA

**Residual Risk**: None (attacker cannot create delegation for victim)

**Severity**: None

---

## 3. Double Usage Recording

### Threat 3.1: Timestamp Collision
**Attack**: Backend submits multiple usage records in same second

**Current Protection**:
-  PDA seeds include timestamp: `[AI_USAGE_SEED, vault.key(), timestamp.to_le_bytes()]`
-  Init constraint prevents duplicate PDA

**Residual Risk**: Only one usage record per vault per second

**Limitation**: Backend could submit multiple records in different seconds

**Mitigation**:
- Backend must implement deduplication logic
- Usage records are immutable (audit trail exists)
- Add sequence number to PDA seed (future enhancement)

**Severity**: Medium (requires malicious backend, limited by vault balance)

---

### Threat 3.2: Replay Attack
**Attack**: Attacker replays old usage transaction

**Current Protection**:
-  Solana's recent blockhash mechanism prevents replay
-  Transaction expires after ~2 minutes

**Residual Risk**: None (Solana protocol protection)

**Severity**: None

---

### Threat 3.3: Cross-Vault Replay
**Attack**: Attacker tries to replay usage from one vault to another

**Current Protection**:
-  PDA seeds include vault.key()
-  Vault ownership checked via `has_one = owner` or delegation

**Residual Risk**: None (vault-specific PDAs)

**Severity**: None

---

### Threat 3.4: Inflated Cost Reporting
**Attack**: Backend submits inflated base_cost to drain user vault

**Current Protection**:
- Warning:  Backend is trusted oracle for base_cost
-  Usage records are immutable (audit trail)
-  Spending limits cap total damage

**Residual Risk**: Backend can inflate costs up to spending limit

**Mitigation**:
- Users should only delegate to trusted backends
- Users can audit usage records vs provider billing
- Admin can change backend_authority if abuse detected
- Implement oracle for provider pricing (future)

**Severity**: Medium (requires malicious backend, limited by spending limit)

---

## 4. Vault Drain

### Threat 4.1: Unauthorized Withdrawal
**Attack**: Attacker tries to withdraw from victim's vault

**Current Protection**:
-  `has_one = owner` constraint
-  Owner must sign transaction

**Residual Risk**: None (cryptographically impossible)

**Severity**: None

---

### Threat 4.2: Rent Drain Attack
**Attack**: Attacker tricks user into withdrawing rent-exempt balance

**Current Protection**:
-  `require!(vault_lamports - amount >= rent_minimum)`
-  Checked before withdrawal

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 4.3: Delegation Drain
**Attack**: Malicious delegate drains vault via excessive usage recording

**Current Protection**:
-  Spending limit caps total damage
-  Expiry limits time window
-  User can revoke instantly

**Residual Risk**: Delegate can drain up to spending limit before user notices

**Mitigation**:
- Users should set conservative spending limits
- Users should use short expiry durations
- Users should monitor vault balance
- Backend should implement rate limiting

**Severity**: Medium (limited by spending limit, requires user to delegate)

---

### Threat 4.4: Fee Vault Confusion
**Attack**: Attacker tries to withdraw from fee_vault by impersonating admin

**Current Protection**:
-  `has_one = authority` on ClaimFees
-  Fee vault PDA seeds verified

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 4.5: Vault Substitution
**Attack**: Attacker provides wrong vault account to charge different user

**Current Protection**:
-  PDA seeds verified: `seeds = [VAULT_SEED, owner.key()]`
-  Ownership constraints: `has_one = owner` or delegation check

**Residual Risk**: None (PDA derivation prevents substitution)

**Severity**: None

---

## 5. Paused State Bypass

### Threat 5.1: Direct Usage Recording
**Attack**: Backend tries to record usage while platform is paused

**Current Protection**:
-  `require!(!config.is_paused)` in both record_ai_usage functions
-  Checked before any state changes

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 5.2: Pause State Race
**Attack**: Backend submits usage transaction, admin pauses, transaction lands

**Current Protection**:
-  Pause check happens during transaction execution
-  If paused when transaction executes, it fails

**Residual Risk**: In-flight transactions may succeed if submitted before pause

**Mitigation**:
- Admin should monitor pending transactions before pausing
- Short transaction expiry (~2 min) limits window

**Severity**: Low (small race window)

---

### Threat 5.3: Pause Bypass via Old Transaction
**Attack**: Attacker pre-signs transaction, waits for unpause, submits old transaction

**Current Protection**:
-  Solana's recent blockhash expires transactions after ~2 minutes
-  Cannot hold transaction indefinitely

**Residual Risk**: None (Solana protocol protection)

**Severity**: None

---

## 6. Model Deactivation Bypass

### Threat 6.1: Deactivated Model Usage
**Attack**: Backend uses deactivated model

**Current Protection**:
-  If model_registry provided: `require!(model_registry.is_active)`
- Warning:  Model registry is optional

**Residual Risk**: Backend can omit model_registry to bypass check

**Mitigation**:
- Make model_registry required (breaking change)
- Or: Enforce backend_authority and trust backend to provide registry
- Current design: Flexible (allows unregistered models)

**Severity**: Low (intentional design for flexibility)

---

### Threat 6.2: Model Registry Substitution
**Attack**: Backend provides wrong model_registry account

**Current Protection**:
-  PDA seeds verified: `seeds = [MODEL_REGISTRY_SEED, model_id.as_bytes()]`
-  model_id in seeds must match parameter

**Residual Risk**: None (PDA derivation prevents substitution)

**Severity**: None

---

### Threat 6.3: Model Deactivation Race
**Attack**: Backend submits usage, admin deactivates model, transaction lands

**Current Protection**:
-  Model active check happens during transaction execution

**Residual Risk**: In-flight transactions may succeed if submitted before deactivation

**Mitigation**:
- Admin should monitor pending transactions
- Short transaction expiry limits window

**Severity**: Low (small race window, limited impact)

---

## 7. Additional Threats

### Threat 7.1: Arithmetic Overflow
**Attack**: Attacker provides values that cause overflow in fee calculation

**Current Protection**:
-  All math uses checked operations (checked_add, checked_sub, checked_mul, checked_div)
-  Transaction panics on overflow

**Residual Risk**: None (fully protected)

**Severity**: None

---

### Threat 7.2: Integer Truncation
**Attack**: Attacker exploits truncation in BPS calculation (e.g., 1 lamport * 500 / 10000 = 0)

**Current Protection**:
-  Calculation uses u128 intermediate: `(amount as u128) * (bps as u128) / 10000`
-  Truncation is intentional (rounds down)

**Residual Risk**: Small amounts may result in 0 fee (acceptable)

**Severity**: None (intentional rounding)

---

### Threat 7.3: Clock Manipulation
**Attack**: Validator manipulates clock to create duplicate usage PDAs

**Current Protection**:
-  Solana clock is consensus-based (requires majority of validators)
-  Clock manipulation is detectable and slashable

**Residual Risk**: Extremely unlikely (requires validator collusion)

**Severity**: None (Solana protocol protection)

---

### Threat 7.4: Account Reinitialization
**Attack**: Attacker tries to reinitialize existing account

**Current Protection**:
-  Anchor's `init` constraint checks discriminator
-  Fails if account already initialized

**Residual Risk**: None (Anchor framework protection)

**Severity**: None

---

### Threat 7.5: PDA Seed Collision
**Attack**: Attacker finds different inputs that produce same PDA

**Current Protection**:
-  SHA256-based PDA derivation (cryptographically secure)
-  Collision probability negligible

**Residual Risk**: None (cryptographic protection)

**Severity**: None

---

### Threat 7.6: Signer Bypass
**Attack**: Attacker provides unsigned account as signer

**Current Protection**:
-  Anchor's `Signer<'info>` type enforces signature
-  Solana runtime verifies signatures

**Residual Risk**: None (Solana protocol protection)

**Severity**: None

---

### Threat 7.7: Account Ownership Bypass
**Attack**: Attacker provides account owned by different program

**Current Protection**:
-  Anchor's `Account<'info, T>` checks program ownership
-  PDA derivation ensures correct program

**Residual Risk**: None (Anchor framework protection)

**Severity**: None

---

## Summary

### Critical Threats (Require Immediate Attention)
None - All critical paths are protected

### High Severity Threats
None - All high-risk attacks are mitigated

### Medium Severity Threats
1. **Admin Fee Manipulation** - Limited by MAX_FEE (20%)
2. **Malicious Backend** - Limited by spending limits, requires user delegation
3. **Timestamp Collision** - Backend can submit multiple records in different seconds
4. **Inflated Cost Reporting** - Backend can inflate costs up to spending limit
5. **Delegation Drain** - Limited by spending limit and expiry

### Low Severity Threats
1. **Revoked Delegation Race** - Small window, limited by spending limit
2. **Pause State Race** - Small window, limited impact
3. **Model Deactivation Race** - Small window, limited impact
4. **Model Deactivation Bypass** - Intentional design (optional registry)

### Mitigated Threats (No Residual Risk)
- Unauthorized fee claiming
- Delegation hijacking
- Expired delegation use
- Spending limit bypass
- Unauthorized withdrawal
- Rent drain
- Paused state bypass
- Arithmetic overflow
- All PDA-based attacks
- All signature-based attacks

---

## Recommendations

### Immediate (Pre-Deployment)
1.  Use multi-sig for admin authority (e.g., Squads)
2.  Use hardware wallet for admin key
3.  Document fee limits and pause policy
4.  Implement backend deduplication logic
5.  Set conservative default spending limits

### Short-Term (Post-Launch)
1. Add sequence number to usage PDA seed (prevent timestamp collision)
2. Enforce backend_authority in delegation usage (stricter control)
3. Add event for authority changes (monitoring)
4. Add event for pause/unpause (monitoring)
5. Implement backend rate limiting

### Long-Term (Future Enhancements)
1. Timelock for admin actions (fee changes, authority transfer)
2. Oracle integration for provider pricing verification
3. Automatic unpause after timeout
4. Grace period for delegation revocation
5. On-chain governance for fee changes

## Incident Response

### Detected Threats
1. **Malicious Backend**: Admin updates backend_authority, users revoke delegations
2. **Admin Key Compromise**: Transfer authority to new multi-sig immediately
3. **Excessive Usage**: Admin pauses platform, investigates, fixes backend
4. **Fee Manipulation**: Community governance to restore reasonable fees
5. **Smart Contract Bug**: Deploy patched version, migrate state if needed

### Monitoring Alerts
- Unusual fee changes (>5% increase)
- Unusual pause/unpause activity
- High-frequency usage from single backend
- Large withdrawals from fee_vault
- Authority changes
- Backend authority changes
- Abnormal vault drain rates
- Failed transaction spikes

### Emergency Procedures
1. **Pause Platform**: `toggle_pause()` - Stops AI usage, preserves user funds
2. **Revoke Backend**: `update_backend_authority()` - Stops malicious backend
3. **Transfer Authority**: `update_authority()` - Recover from key compromise
4. **Communicate**: Notify users via official channels
5. **Investigate**: Analyze on-chain data, identify root cause
6. **Remediate**: Deploy fix, compensate affected users if needed
7. **Post-Mortem**: Document incident, improve security measures
