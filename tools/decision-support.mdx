Decision-support tools exist for workflows where Rabit should be more structured than a normal chat answer.

This is the family for outputs that should feel repeatable, auditable, and easier to trust.

## What this family is for

| Product need | How decision-support tools help |
| --- | --- |
| convert trade ideas into repeatable numbers | sizing tools use deterministic formulas instead of prose guesses |
| compare a tracked market universe quickly | scan tools rank assets from current state instead of arbitrary model preference |
| preserve a post-trade lesson in a structured way | debrief tools write a durable review instead of leaving it inside chat history |

## All decision-support tools

| Tool | Useful for | Value source | Failure shape |
| --- | --- | --- | --- |
| `calculate_position_size` | compute size from entry, stop, and risk budget | pure deterministic formula logic | validation error on invalid prices, side, or missing risk inputs |
| `scan_markets` | rank the tracked asset universe | tracked assets, `market_context`, market service, and live price data | invalid sort or bias raises; per-asset failures are skipped |
| `create_trade_debrief` | store a structured post-trade review | trade debrief service plus authenticated `user_id` | missing user or persistence failure |

## How this family works

```mermaid
flowchart LR
    A[User asks for sizing, ranking, or review] --> B[Decision-support tools]
    B --> C[Deterministic calculation]
    B --> D[Live asset scan]
    B --> E[Structured debrief persistence]
    C --> F[Agent response]
    D --> F
    E --> F
```

## Per-tool breakdown

| Tool | Useful for | How it works | Main output |
| --- | --- | --- | --- |
| `calculate_position_size` | compute a position size from explicit trade parameters | applies deterministic sizing logic to entry, stop, risk budget, and side | structured size calculation |
| `scan_markets` | compare tracked assets quickly | evaluates the tracked asset universe using current market context and scan settings | ranked market list |
| `create_trade_debrief` | save a structured trade review | validates user identity and writes a normalized debrief record to storage | persisted debrief record |

## Why this family exists

| Workflow | Why conversation alone is not enough |
| --- | --- |
| position sizing | users expect the same input to produce the same output |
| market scanning | ranking should come from current signals instead of arbitrary model preference |
| trade debrief | post-trade review becomes more valuable when stored in a consistent structure |

## Error handling and agent behavior

| Tool | Internal handling | What the agent should do |
| --- | --- | --- |
| `calculate_position_size` | hard-fails on invalid inputs such as zero prices or missing risk budget | ask for corrected inputs instead of improvising a number |
| `scan_markets` | validates sorting and bias, but quietly skips individual assets that fail during fetch | return the matched universe and mention that the scan is over tracked assets only |
| `create_trade_debrief` | requires active user identity and valid storage path | explain that debrief persistence requires an authenticated user context |

## Why this family matters

These tools are where Rabit becomes more dependable for real trading workflows.

They reduce the gap between:

- a helpful explanation
- and a structured output the user can actually act on or save
